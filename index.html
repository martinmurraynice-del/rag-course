解决方案演进
  • 方案 1
    模型微调 · SFT（Supervised Fine-Tuning）
    成本高、周期长，更新困难，有数据泄露风险
  • 方案 2
    超长提示词
    消耗大量 token，成本高，检索精度随长度下降
  • 方案 3
    RAG · 检索增强生成 ✨
    外部知识库实时检索，作为 LLM 的额外输入 —— 低成本、高时效、可控

知识库构建详解

知识库构建 Pipeline A – F

从原始数据到可检索的向量索引，离线构建的六大流程

Pipeline A
原始数据采集 · 多源数据的聚合和筛选
🏝️  核心原则：只采集与业务场景强相关的数据，同时记录数据来源

重心在于按照需求场景下的数据分类/分类体系进行采集： 找业务方要 / 让业务方构建 / Python 爬取 / 找高质量下载 / 购买

采集渠道
官网、数据库、API、爬虫、人工录入、采购数据集
数据类型
文档、网页、PDF、图片、音视频、结构化数据
📋 PM 面试真题
▸ 项目中数据来源有哪些？
▸ 数据量级是多大？成本是多少？
Pipeline B
文档解析 · 非结构数据的结构化转化
格式识别 → 格式分类 → 技术匹配 → 技术执行

保留文档原生结构，转化为标准化格式

核心解析技术
OCR · Optical Character Recognition / 光学字符识别 图片提取文本 | 实现过程：版面分析 → 表格识别 → 文本识别 → 图像识别 → 结构整合
ASR · Automated Speech Recognition / 自动语音识别 音频转译文本
TTS · Text To Speech / 语音合成 文字转语音（可用于输出验证环节）
产品经理在此阶段的职责
  • 输出《文档解析质量标准》，明确不同格式数据的解析要求
  • 协调业务文档管理员提供格式说明
  • 定义解析后数据的输出格式
  • 组织抽样校验，推动解决解析异常问题
Pipeline C
数据清洗 · 保留语义核心，剔除无关信息

原始数据中常包含冗余信息（文档页眉页脚、重复段落）、无效内容（乱码、无意义符号），需通过清洗实现「数据瘦身」，同时标准化格式以提升后续处理效率。

剔除冗余信息 → 内容改写 → 格式标准化 → 敏感信息过滤
怎样清洗：质量校验与修正
🤖
自动校验
规则过滤、格式检查
👁️
人工抽样
抽样人工核查质量
🔧
异常处理
标记、修正或丢弃
语义清洗与优化：不只是删除，还要保证清洗后内容语义连贯完整
Pipeline D
文本分割 · Chunk 切片
新词 Chunk：将长文档拆分为语义完整的最小知识单元，是 RAG 检索精度的关键
切分三大原则
A
语义完整性：优先按「自然语义边界」分割（段落、章节、句子），避免将结论与论证过程拆分到不同 Chunk
B
长度适配：根据不同模型、不同使用场景确定切分的具体分割长度
C
重叠窗口机制：相邻 Chunk 保留一定重叠内容，防止语义被硬截断
文本分割 6 大类
固定长度 语义感知 文档结构 关键词/主题 多粒度 混合分块
Pipeline E
文本向量化 · Embedding Model

嵌入模型（Embedding Model）是将文本、图像、音频等数据转换为低维稠密向量的神经网络模型，是 RAG、推荐系统、语义搜索等 AI 应用的核心组件。

输入
任意文本、句子、段落（即 Chunk）
输出
高维浮点数向量（如 768 / 1536 维）
⚠️ 关键：离线构建知识库与在线用户查询，必须使用 同一个 Embedding 模型，才能保证向量空间一致
Pipeline F
向量索引构建 · 专业人士做，了解即可
向量数据库（Vector DB）
通过相似度搜索查找语义相近数据，是 RAG 的核心基础设施
关系型数据库（SQL）
关键词精准匹配，业界最成熟、应用最广泛，适合结构化检索
检索策略对比
策略解决的问题适用场景
混合检索 解决「内容理解」问题（语义 + 关键词） 需要语义理解的问答
结构化搜索 解决「范围限定」问题（时间、类别、权限） 精确过滤、权限控制
两者结合 先粗筛（结构化）→ 再精找（混合）→ 最终排序 企业级知识库
💡 原则：能用结构化检索就不用混合检索，简单优先，降低系统复杂度
02 · 在线过程

检索 Retrieval

由用户触发的在线过程 —— 查询改写、语义检索、多路召回、重排序

检索阶段全景
查询 Query
查询改写和分解；根据文本生成 SQL，在关系型数据库中检索内容
检索 Search
结构化搜索 + 语义搜索；根据文本生成 embedding，在向量数据库中检索内容
索引 Index
快速匹配、定位；嵌入模型必须与离线构建时保持一致
重排 Rerank
Rerank 模型对召回结果二次精排，提升最终结果质量
🎯 场景思维 = 用户故事：每一个检索策略的设计，都要从真实用户使用场景出发
Pipeline G
Query 改写 · 让查询更精准

人工检索时只用当前最新表达，但同一含义有多种说法。Query 改写通过 Seq2Seq 等模型，将原始查询转化为更易被检索系统理解的形式。

六种改写方法
  • 1
    语义强化：通过同义词扩展、语义改写等手段，丰富 query 的语义表达，让检索系统更全面地理解用户意图
  • 2
    歧义消解：识别并消除 query 中的多义词、指代不明等歧义，确保检索方向精准
  • 3
    上下文补全：利用对话历史或会话上下文，将指代词还原为具体内容
  • 4
    意图拆解：将复杂的复合 query 拆分为多个子意图，分别检索后融合，提高召回精度
  • 5
    领域术语适配：将通用表达映射为特定领域的专业术语，提升垂直场景召回率
  • 6
    约束条件强化：显式提取或补充 query 中的时间、地点、数量、格式等约束条件，提升检索过滤精准度
Pipeline H
多路查询 & 查询分解
混合搜索
关键词（BM25）+ 向量检索（Embedding）并行召回
查询构建
将自然语言转化为结构化查询（SQL / 过滤条件）
多路召回 → 结果合并去重 → 重排序（Rerank）→ Top-K 输出
📊 重排序策略：用 score 和 top-K 召回相似度最高的 Chunks，再经 Rerank 模型二次精排后输入 LLM
产品经理职责
  • 明确召回目标（精准率 vs 召回率的权衡）
  • 制定检索质量评估标准（如 Hit Rate、MRR）
